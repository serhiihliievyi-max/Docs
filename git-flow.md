# Git Flow

## Ветки

| Ветка                                                         | Роль                                                       |
| ------------------------------------------------------------- | ---------------------------------------------------------- |
| `main`                                                        | Прод. Будет деплоится на прод автоматически                |
| `dev`                                                         | Предпрод. Сюда идут все PR, здесь тестируем                |
| `feature/[nameUser]/[CAR-[номер таски в линеар]-description]` | Задачи (ветки с новыми фичами с префиксом feature)         |
| `fix/[nameUser]/[CAR-[номер таски в линеар]-description]`     | Баги (ветки которые несут характер правок с префиксом fix) |
| - Если номера таски нету то без номера                        |                                                            |
> [!info] !!!!!!!
> На прямую в main после того как уже заготовака залита, настроенна и т.д. не пушим (то есть какое то время пушат и когда уже начинается командная работу не пушим (в идеале настраивают так что бы в маин только ПР проходил а обычный пуш нет - но мы так заморачиватся не будем)

## Флоу разработчика

### 1. Начало работы — актуализируем staging (checkout переходим в ветку и  командой pull стягиваем с гита то что глобально на нем новое )

```bash
git checkout dev
git pull
```

### 2. Создаём ветку от staging (будучи уже в ветке стейджинг)

```bash
git checkout -b feature/[nameUser]/[CAR-[номер таски в линеар]-description]
```

### 3. Работаем, коммитим

```bash
git add .
git commit -m "feat: добавил форму бронирования"
git push --set-upstream origin feature/[nameUser]/[CAR-[номер таски в линеар]-description]
```

### 4. При долгой задаче — подтягиваем staging (имеется ввиду что пока кто-то работал в одной ветке тем временем уже могли быть сделаны ПР другими участниками потому нужно их изменения перед своим ПР подтянуть в свою ветку с фичей)

```bash
git checkout dev
git pull
git checkout feature/[nameUser]/[CAR-[номер таски в линеар]-description]
git merge dev
git push
```

На этом моменте или все ок или полезут конфликты (если к примеру изменения были кем то внесены в тот файл с котором параллельно работал другой разработчик - тогда ИДЕ в которой кодите будет эти места показывать и нужно будет пофиксить эти накладнки)
### 5. Завершили конфликты → PR в staging

Перед PR обновляем ветку:

```bash
git checkout dev 
git pull
git checkout feature/[nameUser]/[CAR-[номер таски в линеар]-description]
git merge dev
git push
```

Затем:
- Создаём PR (прям на самом гите руками - не через терминал): `feature/... → dev`
- Назначаем ревьюера (если сам себе ревьювер то не обязательно - в больших проектах настраивают что бы к примеру без апрува кем то ПР вообще не прошел)
- Ждём аппрув → мерж

## Релиз в прод

Когда `staging` стабилен и протестирован:

```bash
git checkout main
git pull
git merge dev
git push
```

ПР в меин подхватывается автоматически и делает деплой в прод.

## Hotfix (срочный фикс прода)

```bash
git checkout main
git pull
git checkout -b fix/[nameUser]/[CAR-[номер таски в линеар]-description]

# ... фиксим ...

git push --set-upstream origin fix/[nameUser]/[CAR-[номер таски в линеар]-description]

```

Два PR:

| PR               | Цель                    |
| ---------------- | ----------------------- |
| `fix/... → main` | Срочно в прод           |
| `fix/... → dev`  | Чтобы фикс не потерялся |

---

## Схема

```
feature/... ──PR──→ dev ──merge──→ main (прод)
fix/...     ──PR──→ main + dev
```

## Нейминг коммитов (это не обязательно но можно позаморачиваться )

| Префикс | Когда |
|---------|-------|
| `feat:` | Новая функциональность |
| `fix:` | Исправление бага |
| `docs:` | Изменения в документации |
| `refactor:` | Рефакторинг без изменения функционала |
| `chore:` | Рутина (зависимости, конфиги) |


**Примеры:**
```
feat: добавил карточку машины
fix: исправил расчёт цены брони
docs: обновил API эндпоинты
```
